## Affected Endpoints
- `POST /api/plugins/tunnel-builder/portal-request/` — create tunnel (returns 202 + job_id)
- `GET /api/plugins/tunnel-builder/tunnel-status/<tunnel_id>/` — poll tunnel provisioning status
- `GET /api/plugins/tunnel-builder/psk/<token>/` — one-time PSK retrieval

## Key Interactions to Verify
- Portal submits 5 params (vpn_profile, device, remote_peer_ip, local_network_cidr, protected_network_cidr)
- API validates, creates VPN objects (status=provisioning), queues Celery job
- Celery job maps profile → config params, generates CLI, pushes via SSH, stores PSK
- Portal polls status endpoint until active/failed
- Portal retrieves PSK via one-time URL

## Edge Cases
- Invalid IP/CIDR format → 400
- Non-existent VPNProfile UUID → 404
- Non-existent device → 404
- Device not cisco_ios/cisco_xe → 400
- Duplicate tunnel (device + peer + local + remote) → 409
- SSH timeout → job marks tunnel as failed
- IOS-XE error in send_config_set output → job marks tunnel as failed
- PSK token expired (>15min) → 410
- PSK token already used → 410
- GCM encryption → no integrity value in config
- IKEv1 profile → IKEv1 commands generated
- Concurrent requests for same device → select_for_update serializes

## Critical Paths
- Full happy path: API → Celery job → SSH push → VPN objects active → PSK retrieved
- IKEv2 profile → correct IOS-XE commands generated (algorithm translation verified)
- IKEv1 profile → correct IOS-XE commands generated
- Error path: device unreachable → VPN objects marked failed → portal sees failure
- Error path: IOS-XE rejects command → detected, tunnel marked failed
