## Problem Statement

Network engineers manually SSH into Cisco IOS-XE devices and type 15-30 crypto CLI commands to build IPsec tunnels. This happens monthly, which means engineers forget syntax between uses, copy-paste from old configs, and introduce drift. The existing Nautobot plugin (v0.3.1a3) automates this for internal engineers with full parameter control.

Now a public member-facing portal needs to request tunnel builds programmatically. Portal users are non-technical. They say "I need a tunnel to X" and the system picks all crypto parameters from policy profiles. The portal is a JumpstartPro Rails app with Keycloak OIDC auth.

Two interfaces, one engine: internal form for engineers, REST API for the portal.

## Demand Evidence

- Engineers currently SSH and type commands manually for every tunnel build
- Monthly frequency, but each build is error-prone and takes significant time
- Cross-project pattern already established: VPN portal as thin auth frontend to Nautobot API
- Working alpha plugin (v0.3.1a3) proves the config generation and SSH push works

## Status Quo

Engineers SSH into each Cisco IOS-XE device, type `crypto isakmp policy`, `crypto map`, transform-sets, ACLs... 15-30 commands per tunnel. One typo in a PSK or mismatched DH group and the tunnel doesn't come up. Debugging is another 30 minutes of `show crypto` commands. Because it's monthly, people forget the exact syntax and copy from old configs, introducing configuration drift.

## Target User & Narrowest Wedge

**Internal users:** Network engineers who know IOS-XE crypto config. They want full control over every parameter (IKE version, DH groups, encryption, lifetimes). The existing form serves them.

**Portal users:** Non-technical members who need VPN connectivity. They specify a destination and the system handles everything else. They need the PSK to configure their end.

**Narrowest wedge:** A single REST endpoint that accepts (VPNProfile reference, remote peer IP, device, protected network) and returns a tunnel confirmation with a one-time PSK retrieval URL.

## Constraints

- Must run within Nautobot 3.x plugin framework
- Target devices are Cisco IOS-XE only (cisco_ios / cisco_xe platform drivers)
- SSH credentials from env vars (NAUTOBOT_DEVICE_USERNAME, etc.)
- PSK must not persist in cleartext in JobResult or logs
- Portal authenticates via Nautobot API tokens (service account) or OIDC passthrough
- Existing internal form must continue working unchanged

## Premises

1. **Nautobot VPN models as source of truth.** The job creates VPNTunnel, VPNTunnelEndpoint objects as part of execution. Nautobot owns tunnel state, not just config push.

2. **Custom REST API in the existing plugin.** A proper endpoint at `/api/plugins/tunnel-builder/portal-request/` with a clean request/response contract. Not relying on the generic job execution API.

3. **Config engine stays.** `build_iosxe_policy_config()` remains the config generation core. It gets fed from VPNProfile data instead of raw form fields.

4. **PSK lifecycle: generate, push, retrieve-once.** Nautobot auto-generates the PSK, pushes it to the router, stores it in Nautobot's Secrets framework, and returns a one-time retrieval URL with short TTL. The portal never handles the raw secret. (Prototype: PSK in response directly, flagged as security debt.)

5. **Internal form unchanged.** Engineers keep full manual control via the existing Nautobot form.

## Cross-Model Perspective

An independent Claude subagent reviewed this cold and identified:

- **The real product is the abstraction layer.** VPNProfile curation (which profiles exist, what security posture each represents, who can use which) is the actual product decision. If the profiles are wrong, nothing else matters. Engineering effort should focus disproportionately on profile design and the mapping from profile to config.

- **Challenged premise #4 (PSK in response).** Returning a PSK in an API response means it traverses the network, lands in Rails logging, APM tools, middleware. Secret sprawl from day one. Proposed: one-time retrieval URL pattern. Premise was revised based on this.

- **48-hour prototype:** New `PortalBuildIpsecTunnel` job, 4 inputs, auto-gen PSK, create VPN objects, call existing config engine, push via Netmiko. Test via Nautobot job API. One Rails controller on portal side.

## Approaches Considered

### Approach A: New Job Class Only (Minimal Viable)

Add a `PortalBuildIpsecTunnel` job to the existing plugin. The portal calls Nautobot's built-in job execution API (`POST /api/extras/jobs/<uuid>/run/`). No custom endpoints.

- Effort: S
- Pros: smallest change, zero new dependencies, ships in a day
- Cons: job API returns a JobResult blob (not portal-friendly), PSK lifecycle harder to retrofit, portal must poll for async job completion, no custom validation layer

### Approach B: Custom REST API Viewset (Chosen)

Add a proper REST API endpoint to the plugin. `POST /api/plugins/tunnel-builder/portal-request/` accepts simplified params, handles PSK lifecycle, queues the config push, returns structured response with one-time PSK retrieval URL.

- Effort: M
- Pros: clean portal contract, PSK security built in, synchronous or webhook response, custom validation + rate limiting, OpenAPI docs
- Cons: more code, new API surface to secure

### Approach C: Two Separate Plugins

Keep this plugin for internal ops. Create `nautobot-vpn-portal-api` for the portal-facing API. New plugin imports config generation from this one.

- Effort: L
- Pros: clean separation, independent versioning, single responsibility
- Cons: two packages to maintain, cross-plugin imports fragile on upgrades, overkill for monthly volume

## Recommended Approach

**Approach B: Custom REST API Viewset in the existing plugin.**

The custom endpoint gives a clean contract between the portal and Nautobot. It handles PSK lifecycle properly, provides a portal-friendly response format, and keeps everything in one codebase. The plugin already owns the config generation, SSH push, and internal form. Adding an API surface is a natural extension, not scope creep.

### Key components to build:

1. **`api/` module** — Django REST Framework viewset + serializer
   - `PortalTunnelRequestSerializer`: validates simplified inputs (vpn_profile, remote_peer_ip, device, protected_network_cidr)
   - `PortalTunnelRequestView`: POST endpoint, creates VPN objects, triggers config push, returns response with PSK retrieval URL
   - URL: `/api/plugins/tunnel-builder/portal-request/`

2. **`profile_to_config_params()` mapping function**

   Signature: `profile_to_config_params(vpn_profile: VPNProfile, device: Device, remote_peer_ip: str, protected_network_cidr: str) -> dict`

   Translates VPNProfile fields to the `build_iosxe_policy_config()` input dict. Known mapping challenges:

   | Config param | Source | Notes |
   |---|---|---|
   | `ike_version` | VPNProfile | Direct mapping |
   | `ike_dh_group` | Phase 1 policy `dh_group` | Direct |
   | `ike_lifetime` | Phase 1 policy `sa_lifetime` | Direct |
   | `ikev2_encryption` | Phase 1 policy `encryption_algorithm` | **Value translation needed:** Nautobot uses `"AES-256-CBC"`, config engine expects `"aes-cbc-256"`. Phase 1 policy stores a list (JSONArrayField), select first/highest. |
   | `ikev2_integrity` | Phase 1 policy `authentication_algorithm` | Same translation + list selection |
   | `ipsec_encryption` | Phase 2 policy `encryption_algorithm` | Same translation pattern |
   | `ipsec_integrity` | Phase 2 policy `authentication_algorithm` | Same translation pattern |
   | `ipsec_lifetime` | Phase 2 policy `sa_lifetime` | Direct |
   | `remote_peer_ip` | API request param | Direct |
   | `local_network` | Auto-detect from device primary IP subnet, or API param | |
   | `remote_network` | `protected_network_cidr` from API request | Direct |
   | `wan_interface` | Device interface tagged `role=wan`, or first routed interface | Auto-detect from Nautobot device interfaces |
   | `crypto_map_name` | Auto-generated | `"CRYPTO-MAP"` (convention) |
   | `crypto_map_sequence` | Auto-incremented per device | Query existing VPNTunnel objects for this device, use max(sequence) + 10 |
   | `crypto_acl_name` | Auto-generated | `"VPN-ACL-{sequence}"` |
   | `ipsec_transform_set_name` | Auto-generated | `"IPSEC-TS-{sequence}"` |
   | `ikev2_proposal_name` | Auto-generated | `"IKEv2-PROP-{sequence}"` |
   | `ikev2_policy_name` | Auto-generated | `"IKEv2-POL-{sequence}"` |
   | `ikev2_keyring_name` | Auto-generated | `"IKEv2-KR-{sequence}"` |
   | `ikev2_profile_name` | Auto-generated | `"IKEv2-PROF-{sequence}"` |
   | `isakmp_policy_priority` | Auto-generated (IKEv1 only) | `sequence` value |

   The naming/sequencing params (crypto_map_name, ACL name, transform-set name, IKEv2 names) have no VPNProfile source. They are operational/device-specific. The mapping function auto-generates them using a per-device sequence counter tracked via VPNTunnel objects in Nautobot.

3. **PSK lifecycle**
   - Prototype: `secrets.token_urlsafe(32)`, returned in response body
   - Production: generate PSK, store in Nautobot Secrets, return one-time retrieval endpoint (`GET /api/plugins/tunnel-builder/psk/<token>/`) with 15-minute TTL, single-use

   **PSK retrieval endpoint spec (production):**
   - Token: `secrets.token_urlsafe(48)` (384 bits of entropy)
   - Storage: Django cache backend (Redis/memcached), key = token, value = encrypted PSK, TTL = 900s (15 min)
   - Auth: token-in-URL is sole auth (bearer token in URL path). Rate limit: 10 requests/min per IP.
   - Success: `200 OK` with `{"psk": "..."}`. Token deleted from cache after retrieval (single-use).
   - Expired/used: `410 Gone` with `{"error": "PSK retrieval link has expired or was already used"}`
   - Invalid token: `404 Not Found`

4. **VPN model object creation** — creates `VPNTunnel` + `VPNTunnelEndpoint` objects. Tunnel status tracks provisioning state (`"active"` on success, `"failed"` on error).

   **Known constraint:** `VPNTunnelEndpoint.source_interface` is a `OneToOneField`, meaning one interface can only be assigned to one endpoint. Policy-based IPsec commonly has multiple tunnels on the same WAN interface. Resolution: use `source_ipaddress` (ForeignKey, allows multiple) instead of `source_interface` for the local endpoint. For the remote endpoint, create it with `source_ipaddress` only (remote device is not in Nautobot).

5. **Existing internal form** — unchanged. Engineers continue using it for full manual control.

   **Note:** `forms.py:34` currently filters only `cisco_xe` but help text mentions `cisco_ios`. Fix this filter to include both platform drivers as part of this work.

### Crypto map sequence collision strategy

When multiple tunnels exist on the same device, sequence numbers must not collide. Strategy:
- Query existing `VPNTunnel` objects for the target device
- Extract the highest crypto map sequence in use
- New tunnel gets `max(existing_sequences) + 10` (or 10 if no existing tunnels)
- The sequence is stored on the VPNTunnel object for future lookups

### API behavior: synchronous

The portal API endpoint is **synchronous**. It blocks until the config push completes (typically 10-30 seconds for SSH connect + config push + write mem). The portal should set a timeout of 60 seconds.

Rationale: tunnel builds are infrequent (monthly), the SSH operation is fast, and synchronous responses are simpler for the portal to handle than polling/webhooks.

If the SSH connection takes longer than 45 seconds, the endpoint returns `504 Gateway Timeout`.

### Error responses

| Scenario | HTTP Status | Response body |
|---|---|---|
| Valid request, tunnel built | `201 Created` | `{"tunnel_id": "...", "status": "active", "psk_url": "..."}` |
| Invalid/missing params | `400 Bad Request` | `{"errors": {"field": ["message"]}}` |
| VPNProfile not found | `404 Not Found` | `{"error": "VPN profile not found"}` |
| Device not found | `404 Not Found` | `{"error": "Device not found"}` |
| Device unreachable via SSH | `502 Bad Gateway` | `{"error": "Device unreachable", "detail": "SSH connection failed"}` |
| Config push partial failure | `502 Bad Gateway` | `{"error": "Config push failed", "detail": "..."}` |
| VPN object creation failed | `500 Internal Server Error` | `{"error": "Internal error creating tunnel record"}` |
| SSH timeout | `504 Gateway Timeout` | `{"error": "Device connection timed out"}` |
| Duplicate tunnel (same device + peer) | `409 Conflict` | `{"error": "Tunnel already exists", "tunnel_id": "..."}` |

### What NOT to build (yet):

- Rollback on partial failure (config pushed but VPN objects failed, or vice versa)
- Tunnel teardown / lifecycle management
- Multi-vendor support (Juniper, Palo Alto, etc.)
- Portal-side UI (that's the Rails app's job)
- Approval workflow for tunnel requests

## Success Criteria

- Portal can request a tunnel build with 4 parameters and receive a working tunnel + PSK
- VPN model objects created in Nautobot for every portal-provisioned tunnel
- PSK never persists in cleartext in logs, JobResult, or database (production)
- Internal form continues working unchanged
- Config pushed to device matches what manual CLI would produce
- End-to-end demo: API call -> VPN objects in Nautobot -> config on device -> PSK retrieved

## The Assignment

Map each profile's fields to the parameters `build_iosxe_policy_config()` expects. If the mapping is clean, the architecture works.
